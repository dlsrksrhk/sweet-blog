# DDD Blog 요구사항

## 1. 프로젝트 개요

DDD Blog는 DDD와 TDD를 학습하기 위한 풀스택 블로그 서비스다. 특정 직군이나 분야에 종속되지 않는 범용 블로그를 지향하며, 사용자는 회원가입과 로그인을 통해 글을 작성, 수정, 삭제, 공개할 수 있다.

프로젝트의 주된 목적은 완성도 높은 상용 블로그를 만드는 것이 아니라, 도메인 모델링, 유스케이스 분리, 테스트 주도 개발, 인증/인가, JPA 기반 영속성, Next.js 기반 UI 구현을 체계적으로 연습하는 것이다.

## 2. 기술 스택

### Frontend

- TypeScript
- Next.js
- Markdown 기반 글 작성 에디터
- Cyberpunk 단일 디자인 테마

### Backend

- Java
- Spring Boot
- Spring Security
- JPA
- MySQL
- JWT 인증

### Development

- 학습용 로컬 실행 중심
- 별도 운영 배포 없음
- 테스트 우선 구현 지향
- REST API 우선 설계

## 3. 핵심 목표

- DDD의 계층 구조를 실습한다.
- 도메인 규칙을 엔티티와 값 객체에 배치한다.
- 애플리케이션 서비스에서 유스케이스를 명확히 표현한다.
- TDD로 도메인 규칙과 유스케이스를 검증한다.
- JWT 기반 인증과 기본 보안 정책을 학습한다.
- Markdown, 이미지, 동영상, 코드 블록을 포함하는 글 작성 흐름을 구현한다.

## 4. 주요 사용자

### 비회원

- 공개된 글 목록을 조회할 수 있다.
- 공개된 글 상세를 조회할 수 있다.
- 회원가입을 할 수 있다.
- 로그인을 할 수 있다.

### 회원

- 자신의 프로필 정보를 조회할 수 있다.
- 글을 작성할 수 있다.
- 자신이 작성한 글을 수정할 수 있다.
- 자신이 작성한 글을 삭제할 수 있다.
- 자신이 작성한 글의 공개 상태를 변경할 수 있다.
- 로그아웃할 수 있다.

### 작성자

회원 중 글을 작성한 사용자를 의미한다. 별도의 권한 등급은 아니며, 특정 글에 대한 소유 관계를 표현한다.

### 관리자

초기 버전에서는 관리자 기능을 필수로 구현하지 않는다. 단, 추후 학습 확장을 위해 관리자 권한 개념을 둘 수 있도록 사용자 권한 모델은 확장 가능하게 설계한다.

## 5. 기능 요구사항

## 5.1 회원가입

사용자는 이름, 닉네임, 로그인 ID, 비밀번호를 입력해 회원가입할 수 있다.

### 입력 항목

- 이름
- 닉네임
- 로그인 ID
- 비밀번호

### 규칙

- 로그인 ID는 중복될 수 없다.
- 닉네임은 중복될 수 없다.
- 로그인 ID는 4자 이상 30자 이하로 제한한다.
- 닉네임은 2자 이상 20자 이하로 제한한다.
- 이름은 1자 이상 30자 이하로 제한한다.
- 비밀번호는 8자 이상이어야 한다.
- 비밀번호는 해시 처리 후 저장한다.
- 탈퇴 기능은 초기 버전에서 제외한다.

### 성공 결과

- 새 회원이 생성된다.
- 생성된 회원은 기본 권한 `MEMBER`를 가진다.

### 실패 조건

- 로그인 ID가 이미 존재한다.
- 닉네임이 이미 존재한다.
- 입력값이 규칙을 만족하지 않는다.

## 5.2 로그인

사용자는 로그인 ID와 비밀번호로 로그인할 수 있다.

### 규칙

- 로그인 성공 시 Access Token을 발급한다.
- Refresh Token은 초기 버전에서는 선택 사항으로 둔다.
- Access Token에는 회원 ID와 권한 정보를 포함한다.
- 비밀번호 검증은 해시 비교로 수행한다.

### 성공 결과

- JWT Access Token이 반환된다.
- 프론트엔드는 토큰을 저장하고 인증이 필요한 API 요청에 사용한다.

### 실패 조건

- 로그인 ID가 존재하지 않는다.
- 비밀번호가 일치하지 않는다.
- 계정 상태가 비활성화되어 있다.

## 5.3 로그아웃

사용자는 로그아웃할 수 있다.

### 규칙

- 초기 버전에서는 클라이언트가 Access Token을 삭제하는 방식으로 처리한다.
- Refresh Token을 도입하는 경우 서버에서 Refresh Token을 무효화한다.

## 5.4 내 정보 조회

로그인한 사용자는 자신의 회원 정보를 조회할 수 있다.

### 조회 항목

- 회원 ID
- 이름
- 닉네임
- 로그인 ID
- 권한
- 가입일

### 규칙

- 비밀번호 해시는 응답에 포함하지 않는다.

## 5.5 글 작성

로그인한 사용자는 새 글을 작성할 수 있다.

### 입력 항목

- 제목
- 본문 Markdown
- 요약
- 대표 이미지 URL 또는 업로드 이미지 ID
- 태그 목록
- 공개 상태

### 글 상태

- `DRAFT`: 임시 저장
- `PUBLISHED`: 공개
- `HIDDEN`: 숨김

### 규칙

- 제목은 1자 이상 100자 이하로 제한한다.
- 본문은 비어 있을 수 없다.
- 요약은 300자 이하로 제한한다.
- 태그는 최대 10개까지 지정할 수 있다.
- 태그 이름은 1자 이상 30자 이하로 제한한다.
- 작성자만 글을 수정하거나 삭제할 수 있다.
- 공개 글만 비회원에게 노출된다.

### 성공 결과

- 새 글이 생성된다.
- 작성자가 현재 로그인한 회원으로 지정된다.

## 5.6 글 수정

작성자는 자신이 작성한 글을 수정할 수 있다.

### 수정 가능 항목

- 제목
- 본문 Markdown
- 요약
- 대표 이미지
- 태그
- 공개 상태

### 규칙

- 작성자가 아닌 사용자는 수정할 수 없다.
- 삭제된 글은 수정할 수 없다.
- 공개 상태 변경은 작성자가 직접 수행한다.

## 5.7 글 삭제

작성자는 자신이 작성한 글을 삭제할 수 있다.

### 규칙

- 초기 버전에서는 소프트 삭제를 사용한다.
- 삭제된 글은 목록과 상세 조회에서 제외한다.
- 작성자가 아닌 사용자는 삭제할 수 없다.

## 5.8 글 목록 조회

사용자는 공개된 글 목록을 조회할 수 있다.

### 조회 조건

- 페이지 번호
- 페이지 크기
- 키워드
- 태그
- 작성자 닉네임

### 정렬

- 최신순
- 오래된순
- 조회수순

### 응답 항목

- 글 ID
- 제목
- 요약
- 작성자 닉네임
- 대표 이미지
- 태그 목록
- 조회수
- 공개일
- 수정일

### 규칙

- 비회원과 일반 회원은 공개된 글만 조회할 수 있다.
- 작성자는 별도의 내 글 목록에서 자신의 임시 저장 글과 숨김 글을 조회할 수 있다.

## 5.9 글 상세 조회

사용자는 공개된 글 상세를 조회할 수 있다.

### 응답 항목

- 글 ID
- 제목
- 본문 Markdown
- 렌더링된 HTML은 프론트엔드에서 생성
- 작성자 정보
- 태그 목록
- 대표 이미지
- 조회수
- 공개일
- 수정일

### 규칙

- 공개 글은 누구나 조회할 수 있다.
- 임시 저장 글과 숨김 글은 작성자만 조회할 수 있다.
- 상세 조회 시 조회수가 증가한다.
- 같은 사용자의 반복 조회수 제한은 초기 버전에서 제외한다.

## 5.10 내 글 관리

로그인한 사용자는 자신이 작성한 글을 관리할 수 있다.

### 기능

- 내 글 목록 조회
- 임시 저장 글 조회
- 공개 글 조회
- 숨김 글 조회
- 글 수정
- 글 삭제
- 공개 상태 변경

## 5.11 Markdown 에디터

글 작성 화면은 Markdown 작성과 미리보기를 지원한다.

### 필수 기능

- Markdown 입력
- 실시간 또는 탭 기반 미리보기
- 제목 입력
- 요약 입력
- 태그 입력
- 코드 블록 입력
- 이미지 삽입
- 동영상 임베드 삽입
- 임시 저장
- 공개 발행

### Markdown 지원

- 제목
- 문단
- 인용문
- 목록
- 링크
- 이미지
- 코드 블록
- 표
- 구분선

### 코드 블록

- 언어명을 지정할 수 있어야 한다.
- 프론트엔드에서 구문 강조를 적용한다.

### 동영상

- 초기 버전에서는 YouTube 등 외부 동영상 URL 임베드를 지원한다.
- 로컬 동영상 파일 업로드는 선택 사항으로 둔다.

## 5.12 이미지 업로드

사용자는 글 본문 또는 대표 이미지로 이미지를 사용할 수 있다.

### 초기 구현 옵션

- 로컬 파일 시스템 저장
- 또는 MySQL에는 메타데이터만 저장하고 파일은 로컬 디렉터리에 저장

### 규칙

- 허용 확장자는 `jpg`, `jpeg`, `png`, `webp`, `gif`로 제한한다.
- 파일 크기 제한을 둔다.
- 업로드한 파일의 원본 파일명과 저장 파일명을 분리한다.
- 이미지 URL을 통해 프론트엔드에서 표시할 수 있어야 한다.

## 5.13 태그

글은 여러 태그를 가질 수 있다.

### 규칙

- 태그 이름은 중복 저장하지 않는다.
- 대소문자 정책은 초기 버전에서 소문자 정규화로 처리한다.
- 글 하나에는 최대 10개의 태그를 연결할 수 있다.

## 5.14 검색

사용자는 글 제목, 요약, 본문을 기준으로 공개 글을 검색할 수 있다.

### 초기 범위

- 단순 키워드 검색
- 제목과 요약 우선
- 본문 포함 검색은 선택 사항

### 추후 확장

- 전문 검색
- 태그 조합 검색
- 작성자 검색

## 5.15 댓글

사용자는 공개된 글에 댓글을 작성할 수 있다.

### 댓글 작성자 유형

- 회원 댓글: 로그인한 회원이 작성한 댓글
- 익명 댓글: 로그인하지 않은 사용자가 닉네임과 임시 비밀번호를 입력해 작성한 댓글

### 입력 항목

#### 회원 댓글

- 본문

#### 익명 댓글

- 닉네임
- 임시 비밀번호
- 본문

### 규칙

- 댓글은 특정 글 하나에 속한다.
- 댓글 본문은 비어 있을 수 없다.
- 회원 댓글은 작성한 회원만 수정할 수 있다.
- 익명 댓글은 작성 시 입력한 임시 비밀번호가 맞을 때만 수정할 수 있다.
- 익명 댓글의 임시 비밀번호는 평문으로 저장하지 않고 해시로 저장한다.
- 초기 버전에서는 댓글 삭제, 대댓글, 신고, 관리자 숨김 처리를 제외한다.

### 성공 결과

- 댓글이 생성된다.
- 생성된 댓글은 글 상세 화면에서 조회할 수 있다.

### 실패 조건

- 댓글 본문이 비어 있다.
- 익명 댓글 작성 시 닉네임이 비어 있다.
- 익명 댓글 작성 시 임시 비밀번호가 비어 있다.
- 회원 댓글 수정 요청자가 작성자가 아니다.
- 익명 댓글 수정 시 임시 비밀번호가 일치하지 않는다.

## 6. 비기능 요구사항

## 6.1 보안

학습용 프로젝트이지만 다음 보안 정책은 구현한다.

- 비밀번호는 BCrypt 등 안전한 해시 알고리즘으로 저장한다.
- JWT 서명 키는 환경 변수 또는 설정 파일에서 관리한다.
- 인증이 필요한 API는 토큰 검증을 수행한다.
- 글 수정, 삭제, 숨김 조회는 작성자 권한을 검증한다.
- 클라이언트 입력값은 서버에서 다시 검증한다.
- Markdown 렌더링 결과는 XSS 방지를 위해 sanitize한다.
- 이미지 업로드 시 확장자와 MIME 타입을 검증한다.
- CORS는 로컬 프론트엔드 주소만 허용한다.
- 운영 수준의 세밀한 보안 정책은 제외하되, 왜 제외했는지 문서화한다.

## 6.2 성능

- 글 목록은 페이지네이션을 사용한다.
- 대량 태그 조회에서 N+1 문제가 발생하지 않도록 테스트하거나 쿼리를 확인한다.
- 상세 조회의 조회수 증가는 단순 동기 처리로 시작한다.

## 6.3 접근성

- 주요 버튼과 입력 필드에 명확한 레이블을 제공한다.
- 키보드로 로그인, 글 작성, 글 발행을 수행할 수 있어야 한다.
- 색상 대비는 사이버펑크 테마 안에서도 읽기 가능한 수준을 유지한다.

## 6.4 디자인

### 테마 방향

- 단일 Cyberpunk 스타일
- 어두운 배경
- 네온 계열 강조색
- 높은 대비
- 글 읽기 영역은 과한 장식을 줄이고 가독성을 우선

### 주요 화면

- 공개 글 목록
- 글 상세
- 회원가입
- 로그인
- 내 글 관리
- 글 작성/수정 에디터
- 내 정보

### 디자인 원칙

- 전체 분위기는 사이버펑크지만 글 읽기 경험을 해치지 않는다.
- 버튼, 입력창, 카드, 태그는 일관된 네온 스타일을 사용한다.
- 라이트 테마는 지원하지 않는다.

## 7. DDD 설계 방향

## 7.1 Bounded Context

### Member Context

회원가입, 로그인, 회원 정보 조회, 권한을 담당한다.

### Blog Context

글 작성, 수정, 삭제, 공개 상태, 태그, 조회를 담당한다.

### Media Context

이미지 업로드와 파일 메타데이터를 담당한다.

### Auth Context

JWT 발급, 토큰 검증, 인증된 사용자 식별을 담당한다.

초기 구현에서는 Auth Context를 Member Context 안의 애플리케이션 서비스로 시작할 수 있다. 이후 학습 목적에 따라 분리한다.

## 7.4 도메인 규칙 후보

- 회원은 중복된 로그인 ID를 가질 수 없다.
- 회원은 중복된 닉네임을 가질 수 없다.
- 글은 작성자 없이는 존재할 수 없다.
- 글 제목은 비어 있을 수 없다.
- 글 본문은 비어 있을 수 없다.
- 작성자만 글을 수정할 수 있다.
- 작성자만 글을 삭제할 수 있다.
- 삭제된 글은 다시 수정할 수 없다.
- 공개되지 않은 글은 작성자만 볼 수 있다.
- 글 하나에는 최대 10개의 태그만 연결할 수 있다.

중복 검사처럼 저장소 조회가 필요한 규칙은 도메인 서비스 또는 애플리케이션 서비스에서 처리한다.

## 8. TDD 학습 계획

## 8.1 테스트 우선순위

2. 도메인 엔티티 규칙 테스트

4. Repository 통합 테스트
5. Controller API 테스트
6. 프론트엔드 컴포넌트 및 사용자 흐름 테스트

## 8.2 백엔드 테스트 예시

### 회원가입

- 유효한 입력이면 회원이 생성된다.
- 중복 로그인 ID면 실패한다.
- 중복 닉네임이면 실패한다.
- 비밀번호는 평문으로 저장되지 않는다.

### 로그인

- 올바른 로그인 ID와 비밀번호면 토큰이 발급된다.
- 비밀번호가 틀리면 실패한다.
- 존재하지 않는 로그인 ID면 실패한다.

### 글 작성

- 로그인한 회원은 글을 작성할 수 있다.
- 제목이 비어 있으면 실패한다.
- 본문이 비어 있으면 실패한다.
- 태그가 10개를 넘으면 실패한다.

### 글 수정

- 작성자는 자신의 글을 수정할 수 있다.
- 작성자가 아닌 회원은 수정할 수 없다.
- 삭제된 글은 수정할 수 없다.

### 글 조회

- 공개 글은 비회원도 조회할 수 있다.
- 임시 저장 글은 작성자만 조회할 수 있다.
- 상세 조회 시 조회수가 증가한다.

## 8.3 프론트엔드 테스트 예시

- 회원가입 폼은 필수 입력값을 검증한다.
- 로그인 성공 시 인증 상태가 갱신된다.
- 글 작성 화면에서 Markdown 미리보기가 표시된다.
- 태그 입력은 최대 10개로 제한된다.
- 비로그인 사용자는 글 작성 화면에 접근할 수 없다.

## 11. 화면 요구사항

## 11.1 공개 글 목록

- 최신 공개 글 목록을 표시한다.
- 검색어와 태그로 필터링할 수 있다.
- 페이지네이션을 제공한다.
- 글 카드에는 제목, 요약, 작성자, 태그, 공개일을 표시한다.

## 11.2 글 상세

- 제목, 작성자, 공개일, 수정일, 본문, 태그를 표시한다.
- Markdown 본문을 HTML로 렌더링한다.
- 코드 블록은 구문 강조를 적용한다.
- 작성자인 경우 수정과 삭제 버튼을 표시한다.

## 11.3 회원가입

- 이름, 닉네임, 로그인 ID, 비밀번호를 입력한다.
- 입력값 오류를 즉시 또는 제출 시 표시한다.
- 성공 시 로그인 화면 또는 메인 화면으로 이동한다.

## 11.4 로그인

- 로그인 ID와 비밀번호를 입력한다.
- 성공 시 메인 화면으로 이동한다.
- 실패 시 오류 메시지를 표시한다.

## 11.5 내 글 관리

- 내 글을 상태별로 조회한다.
- 임시 저장, 공개, 숨김 글을 구분한다.
- 각 글에서 수정, 삭제, 상태 변경을 수행할 수 있다.

## 11.6 글 작성/수정

- 제목 입력 영역
- Markdown 본문 에디터
- 미리보기 영역
- 요약 입력
- 태그 입력
- 대표 이미지 설정
- 이미지 삽입
- 동영상 임베드 삽입
- 임시 저장 버튼
- 공개 발행 버튼

## 15. 성공 기준

프로젝트 1차 완성 기준은 다음과 같다.

- 사용자가 회원가입할 수 있다.
- 사용자가 로그인하고 JWT를 받을 수 있다.
- 인증된 사용자가 글을 작성할 수 있다.
- 작성자가 자신의 글을 수정하고 삭제할 수 있다.
- 비회원이 공개 글 목록과 상세를 조회할 수 있다.
- 비공개 글은 작성자만 조회할 수 있다.
- Markdown 본문이 글 상세에서 정상 렌더링된다.
- 주요 도메인 규칙에 대한 단위 테스트가 존재한다.
- 주요 유스케이스에 대한 서비스 테스트가 존재한다.
- API 테스트 또는 통합 테스트로 핵심 흐름이 검증된다.
