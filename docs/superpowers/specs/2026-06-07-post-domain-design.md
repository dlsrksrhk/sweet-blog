# 글 작성 도메인 1차 설계

## 1. 배경

DDD Blog는 Spring Boot 3.5, Java 21, JPA, MySQL, Next.js, TypeScript를 사용해 DDD와 TDD를 학습하기 위한 블로그 프로젝트다.

현재 백엔드는 `backend` 폴더에 Spring Boot 프로젝트가 생성되어 있고, `PostTitle` 값 객체와 단위 테스트가 존재한다. 다음 단계는 JPA, Repository, 애플리케이션 서비스, API를 붙이기 전에 순수 Java 도메인 객체와 JUnit 테스트만으로 글 작성 도메인의 첫 번째 조각을 구현하는 것이다.

이번 설계는 작은 범위를 테스트로 검증하면서 진행하는 것을 목표로 한다. 영속성, 웹, 인증 관심사는 이번 단계에서 제외하고 도메인 규칙에 집중한다.

## 2. 구현 범위

이번 작업에 포함하는 내용은 다음과 같다.

- 백엔드 테스트 메서드명을 한글 시나리오형으로 작성한다는 프로젝트 규칙 추가
- 기존 `PostTitleTest` 테스트 메서드명을 한글로 변경
- `PostContent` 값 객체 추가
- `PostSummary` 값 객체 추가
- `TagName` 값 객체 추가
- `AuthorId` 값 객체 추가
- `PostStatus` enum 추가
- `Post` aggregate 추가
- 새 도메인 규칙에 대한 단위 테스트 추가
- `docs/requirements.md`에 댓글 요구사항 추가

이번 작업에서 제외하는 내용은 다음과 같다.

- JPA 엔티티 매핑
- Repository 인터페이스와 구현체
- 애플리케이션 서비스
- REST API
- 인증 연동
- MySQL 설정
- 프론트엔드 구현
- 댓글 도메인 구현

댓글은 이번에 요구사항 문서에는 추가하지만, 실제 구현은 별도 설계와 구현 단계에서 진행한다.

## 3. 도메인 모델

### 3.1 Post

`Post`는 블로그 글 하나를 표현하는 aggregate다. 이번 단계에서는 글 생성 시점의 규칙을 책임진다.

필드:

- `AuthorId authorId`
- `PostTitle title`
- `PostContent content`
- `PostSummary summary`
- `List<TagName> tags`
- `PostStatus status`

규칙:

- 글은 작성자 없이 생성될 수 없다.
- 글은 제목 없이 생성될 수 없다.
- 글은 본문 없이 생성될 수 없다.
- 글은 상태 없이 생성될 수 없다.
- 글 하나에는 태그를 0개부터 10개까지 지정할 수 있다.
- `Post`는 전달받은 태그 목록을 방어적으로 복사한다.
- 외부에 노출되는 태그 목록은 호출자가 수정할 수 없어야 한다.

이번 단계에서는 글 ID를 넣지 않는다. 글 ID는 JPA와 Repository를 설계할 때 추가한다. 아직 저장하지 않는 순수 도메인 모델이기 때문이다.

### 3.2 PostContent

`PostContent`는 Markdown 본문을 표현한다.

규칙:

- 본문은 `null`일 수 없다.
- 본문은 blank일 수 없다.
- 본문 문자열은 입력값 그대로 보관한다.

### 3.3 PostSummary

`PostSummary`는 선택 가능한 짧은 요약을 표현한다.

규칙:

- `null`은 빈 요약으로 정규화한다.
- blank 문자열도 빈 요약으로 정규화한다.
- blank가 아닌 문자열은 앞뒤 공백을 제거한다.
- 정규화 후 요약은 300자 이하여야 한다.

이 정책을 사용하면 에디터에서 요약 입력칸을 비워둔 상태로 저장하는 흐름을 자연스럽게 처리할 수 있다.

### 3.4 TagName

`TagName`은 정규화된 태그명 하나를 표현한다.

규칙:

- 태그명은 `null`일 수 없다.
- 태그명은 blank일 수 없다.
- 태그명은 앞뒤 공백을 제거하고 소문자로 정규화한다.
- 정규화된 태그명은 30자 이하여야 한다.

중복 태그 처리 정책은 이번 단계에서 제외한다. 태그 저장 방식이 정해지는 Repository 또는 애플리케이션 서비스 설계 단계에서 다시 다룬다.

### 3.5 AuthorId

`AuthorId`는 글 작성자인 회원의 식별자를 표현한다.

규칙:

- 작성자 ID는 `null`일 수 없다.

초기 구현에서는 내부 값으로 `Long`을 사용한다. 회원 저장 방식이 바뀌더라도 이 값 객체가 변경 지점을 감싸도록 한다.

### 3.6 PostStatus

`PostStatus`는 글의 공개 상태를 표현한다.

상태:

- `DRAFT`: 임시 저장
- `PUBLISHED`: 공개
- `HIDDEN`: 숨김

상태 전이 규칙은 이번 단계에서 제외한다. 이번 단계에서는 글 생성 시 상태가 반드시 있어야 한다는 규칙만 검증한다.

## 4. 댓글 요구사항

기존 요구사항 문서에서는 댓글이 제외 범위에 들어가 있었다. 이제 댓글 기능을 요구사항에 포함한다.

새 댓글 요구사항:

- 댓글은 특정 글 하나에 속한다.
- 댓글은 로그인한 회원이 작성할 수 있다.
- 댓글은 익명 사용자도 작성할 수 있다.
- 익명 댓글 작성 시 표시 닉네임과 임시 비밀번호를 입력한다.
- 익명 댓글의 임시 비밀번호는 평문으로 저장하지 않고 해시로 저장한다.
- 회원 댓글은 작성한 회원만 수정할 수 있다.
- 익명 댓글은 작성 시 입력한 임시 비밀번호가 맞을 때만 수정할 수 있다.
- 댓글 본문은 비어 있을 수 없다.

댓글 초기 범위에서 제외하는 내용:

- 댓글 삭제
- 대댓글
- 댓글 신고
- 관리자 댓글 관리
- 댓글 숨김 처리

댓글은 댓글 도메인 구현 단계에서 별도의 `Comment` aggregate로 설계한다. 댓글을 `Post` 안의 단순 컬렉션으로 넣지 않는다. 글과 댓글은 생명주기와 권한 규칙이 다르기 때문이다.

## 5. 프로젝트 규칙

루트 `AGENTS.md`에 다음 규칙을 추가한다.

> 백엔드 테스트 메서드명은 IntelliJ 테스트 결과 목록에서 한국어 행동 설명으로 읽히도록 한글 시나리오형으로 작성한다.

예시:

- `제목을_생성한다`
- `본문이_비어_있으면_생성할_수_없다`
- `태그명은_소문자로_정규화된다`
- `태그가_10개를_초과하면_글을_생성할_수_없다`

이 규칙은 새로 작성하는 백엔드 테스트와, 이번 작업에서 수정하는 기존 테스트에 적용한다.

## 6. 테스트 계획

모든 구현은 단위 테스트를 기준으로 진행한다.

필수 테스트:

- `PostTitleTest`
  - 기존 테스트 메서드명을 한글로 변경한다.
- `PostContentTest`
  - 유효한 본문을 생성한다.
  - `null` 본문은 거부한다.
  - blank 본문은 거부한다.
- `PostSummaryTest`
  - `null` 요약은 빈 요약이 된다.
  - blank 요약은 빈 요약이 된다.
  - blank가 아닌 요약은 앞뒤 공백이 제거된다.
  - 300자를 초과하는 요약은 거부한다.
  - 정확히 300자인 요약은 허용한다.
- `TagNameTest`
  - 유효한 태그명을 생성한다.
  - 태그명은 앞뒤 공백 제거 후 소문자로 정규화된다.
  - `null` 태그명은 거부한다.
  - blank 태그명은 거부한다.
  - 30자를 초과하는 태그명은 거부한다.
- `AuthorIdTest`
  - 유효한 작성자 ID를 생성한다.
  - `null` 작성자 ID는 거부한다.
- `PostTest`
  - 유효한 글을 생성한다.
  - 작성자가 없으면 글을 생성할 수 없다.
  - 제목이 없으면 글을 생성할 수 없다.
  - 본문이 없으면 글을 생성할 수 없다.
  - 상태가 없으면 글을 생성할 수 없다.
  - 태그 10개는 허용한다.
  - 태그가 10개를 초과하면 글을 생성할 수 없다.
  - 외부에 노출된 태그 목록은 수정할 수 없다.

검증 명령:

```powershell
cd C:\dev\study\ddd-blog\backend
$env:JAVA_HOME='C:\java\jdk-21'
.\gradlew.bat test
```

기대 결과:

- Gradle test task가 통과한다.
- IntelliJ 테스트 목록에 한글 테스트 메서드명이 표시된다.

## 7. 오류 처리

이번 단계의 도메인 모델은 잘못된 입력에 대해 `IllegalArgumentException`을 던진다.

커스텀 도메인 예외와 에러 코드는 애플리케이션/API 계층을 설계할 때 도입 여부를 결정한다. 이번 단계에서는 도메인 규칙 검증에 집중하기 위해 예외 체계를 단순하게 유지한다.

## 8. 구현 메모

- 새 도메인 클래스는 `backend/src/main/java/com/dddblog/backend/blog/domain` 아래에 둔다.
- 새 테스트는 `backend/src/test/java/com/dddblog/backend/blog/domain` 아래에 둔다.
- 도메인 클래스에는 Spring annotation을 붙이지 않는다.
- 도메인 클래스에는 당장 Lombok을 사용하지 않는다.
- 값 객체는 불변으로 만든다.
- 컬렉션은 방어적으로 복사한다.
- 예외 메시지는 짧고 일관되게 유지한다.
